软件构造基础

对软件的视图：

按阶段划分：构造时/运行时视图

按动态性划分：时刻/阶段视图

按构造对象的层次划分：代码/构件视图

版本管理：vcs（集中管理）、git

质量影响因素：外部质量因素：影响、用户

内部质量因素影响：软件本身和它的开发者

外部质量取决于内部质量

正确性：最重要的质量指标

调试时：发现不正确要进行消除。

防御性编程：在写程序时就保持正确性

形式化方法：通过形式化验证发现问题

健壮性：针对异常情况的处理

  1. 健壮性是对正确性的补充
  2. 正确性：软件的行为要严格的符合规约中定义的行为
  3. 健壮性：出现规约定义之外的情形的时候，软件要做出恰当的反应

五大质量

健壮性（chapter7）——对于正确性的补充，处理spec中没有明确的情况
可拓展性（chapter6）——应对变化的能力（可维护性）主观地判断
可复用性（chapter5）——1.一次开发 2.多次使用
可理解性 (chapter4）
效率（chapter8） 性能必须建立在正确性的基础上（过早的优化是万恶之源）

测试

再好的测试也无法证明系统里不存在错误

静态测试：Reviews, walkthroughs, or inspections

动态测试：

测试：发现是否存在错误

调试：识别错误根源，消除错误

测试用例：输入+执行条件+期望结果

Spec describes the input and output behavior of the function

先写测试会节省大量的调试时间

等价类划分:

基于等价类划分的测试：将被测函数的输入域划分为等价类，从等价类中导出测试用例

针对每个输入数据需要满足的约束条件，划分等价类

每个等价类代表着对输入约束加以满足/违反的有效/无效数据的集合

基于的假设：相似的输入，将会展示相似的行为。故可从每个等价类中选一个代表作为测试用例即可

从而可以降低测试用例数量

需要考虑输入数据的特殊情况

考虑输入的上限：很大的数是否仍正确？

大量的错误发生在输入域的“边界”而非中央，边界值分析方法是对等价类划分方法的补充

独立/基本路径测试：对程序所有执行路径进行等价类划分，找出有代表性的最简单的路径(例如循环只需执行1次)，设计测试用例使每一条基本路径被至少覆盖1次

代码覆盖度：已有的测试用例有多大程度覆盖了被测程序

路径覆盖>分支覆盖>语句覆盖

回归测试：一旦程序被修改，重新执行之前的所有测试

测试策略（根据什么来选择测试用例）非常重要，需要在程序中显式记录下来

目的：在代码评审过程中，其他人可以理解你的测试，并评判你的测试是否足够充分
